# Development Plan — 2026-05-01

Strategic direction and bead roadmap following a backlog review and comparison with
the Hangar fleet management tool.

---

## Context

FleetRoll was built to make managing the Linux moonshot fleet easier — specifically
diagnosing and recovering hung hosts, tracking puppet/override/vault state, and
eventually orchestrating rollouts. Hangar covers higher-level fleet capacity and pool
health but doesn't do override management, rollout orchestration, or deep per-host
health signal. These remain fleetroll's differentiated value.

---

## Priority tracks

### 1. More signal — build a better picture of machine state [P2]

**Goal:** Replace ad-hoc filter queries (e.g. `moonshots-hung`) with first-class
derived health signals backed by richer data collection.

**Rationale:** The hung-machine reboot tool works but is brittle — it fires based on
a manual filter query rather than a semantic `HUNG` field. Adding structured signals
makes automated remediation and rollout validation possible.

- `mvp-z4lh` gather_host: integrate gwhc health check data (Linux) — optional
  `sudo gwhc --json`, fields prefixed `gwhc_*`. Closes `mvp-qg2`, partially closes
  `mvp-gpke`.
- `mvp-6eri` gather_tc: collect per-host TC job success history — last N jobs /
  last 24h pass/fail rate. Data layer for hung detection and rollout validation.
  Feeds `mvp-2ip`.
- `mvp-3jg8` derive `HUNG` status from gwhc state + TC job history — host is HUNG
  if worker processes are running and IDLE but no completed TC jobs in configurable
  window. Depends on `mvp-z4lh`, `mvp-6eri`.

**Other related open beads:** `mvp-2ip` (RO_HEALTH uses TC job history), `mvp-05c`
(ping), `mvp-194` (TC quarantine reason), `mvp-5qd` (SIP status on macOS).

**Build order:** `mvp-z4lh` + `mvp-6eri` in parallel → `mvp-3jg8`

---

### 2. Rollouts — the original goal [P2]

**Goal:** Orchestrate override rollouts across host populations in waves, with
automatic safety gates that stop a bad rollout before it propagates.

**Rationale:** Primary differentiator vs Hangar and the original motivation for the
tool. Bulk override management (`mvp-3vc.1`, `mvp-t8yk`) is the tactical building
block; wave-based rollout with post-deploy validation is the strategic goal.

- `mvp-3vc` epic: rollout orchestration
- `mvp-hc36` TUI/web: override + vault picker — list locally available files at
  deploy time so the operator doesn't need to know the SHA prefix out of band (no
  more `deploy_vault_yaml_e18921a45ae8.sh`-style scripts). Natural entry point for
  "start a rollout with this override." Integrates into TUI edit-mode action menu
  (`mvp-26z`) and web UI. Related: `mvp-kaw5`, `mvp-3vc`.
- `mvp-3vc.1` host-unset-override across a population (filter query)
- `mvp-t8yk` host-unset-override by commit SHA — uses `override_sha256` already in
  the audit log, no new metadata needed
- `mvp-osyq` rollout post-deploy wave validation — check TC job success rate on
  rolled-out hosts before advancing wave. Core safety gate. Depends on `mvp-6eri`,
  `mvp-3vc`.

**Build order:** `mvp-hc36` is a natural first step (usable standalone before full
rollout orchestration exists); `mvp-6eri` → `mvp-osyq`; `mvp-3vc.1` + `mvp-t8yk`
can proceed in parallel.

---

### 3. Central read-only web UI — rollout status communication [P1, deferred]

**Goal:** Give coworkers a browser-based view of fleet state and rollout progress
without needing CLI access.

**Rationale:** Once rollouts exist, operators need to communicate rollout status to
the wider team. The local write-capable UI (`mvp-kaw5`) is already functional for
personal use. The central read-only dashboard (`mvp-si5u`) is the team-facing layer.

**Sequencing:** Deliberately last. Investing in the web UI before the data and
rollout layers are solid means building a dashboard for data that doesn't exist yet.
The web UI will consume track 1 + 2 signals automatically once they're in the DB.

- `mvp-si5u.2` Phase 2: read-only central server (FastAPI + Postgres + SSE) —
  highest-leverage unlocker, blocks 4 downstream beads, P1.
- `mvp-si5u.3` Phase 2.5: split web UI into local write-capable vs central
  read-only.
- `mvp-kaw5.4` auth: implement loopback-only invariant (small, well-scoped).
